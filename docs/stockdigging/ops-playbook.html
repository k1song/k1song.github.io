1. 시스템 전체 구조

StockDigging 전체 데이터 흐름을 한눈에 파악할 수 있는 구조도이다. 모든 데이터는 왼쪽에서 오른쪽으로 흐른다.

[데이터 소스] [서버: 158.247.235.58] [사용자] /var/www/stockdigging/ DART API ──> collectors/ ──> stockpulse.db ──> scoring.py ──> score_cache.pkl ──> API ──> 프론트엔드 pykrx ──> (KR 가격) (2.1GB) (1,907줄) (17MB) (109개) FMP ────> (US 가격+밸류에이션) FMP ────> (US 재무) [크론 자동화] 16:00 KST 매일(월~토) ──> auto_daily.sh (KR: 가격 > 기술 > 밸류에이션 > 대회 > 캐시) 06:00 KST 매일(월~금) ──> auto_daily_us.sh (US: FMP가격 > 기술 > FMP밸류 > 캐시) 매월 1일 00:00 UTC ────> auto_monthly.sh (종목목록 > 캘리브레이션) 분기 (8회/년) ─────────> auto_quarterly.sh (재무제표 > 캘리브레이션) [nginx] :443 (HTTPS) ──> proxy_pass :8000 ──> uvicorn (quick_server.py) 유지보수 시 return 503 전환
핵심 포인트
scoring.py가 DB 데이터를 읽어 score_cache.pkl을 생성하고, API가 이 캐시를 서빙한다. 배치 완료 후 /api/cache/refresh를 호출하여 서버 재시작 없이 캐시를 자동 갱신한다.

4. auto_daily.sh 상세 (KR 전용, 6단계)

매일 크론이 실행하는 KR 일일 배치의 6단계 흐름이다.

단계작업스크립트소요 시간
1KR 가격 수집run_batch.py --market KR --step price~3분
2KR 기술적 지표run_batch.py --market KR --step technical~2분
3KR 밸류에이션 계산calc_kr_valuations.py~2분
4모의투자 대회 스냅샷update_competition.py~1분
5사전등록 시뮬레이션curl /admin/run-pending-simulations~1분
6스코어 캐시 갱신curl /api/cache/refresh백그라운드 ~5분
중요 변경 (2026-03-26)
Naver Finance API / merge 스크립트는 제거되었다. KR 밸류에이션은 calc_kr_valuations.py(DART 재무 + 종가 기반 자체 계산)만 사용한다.
캐시 정책
배치 완료 후 /api/cache/refresh를 호출하여 서버 재시작 없이 캐시를 갱신한다. 캐시를 수동으로 삭제해야 하는 경우는 스코어링 코드(scoring.py)를 변경했을 때뿐이다.

4-2. auto_daily_us.sh 상세 (US 전용, FMP 기반, 4단계)

매일 크론이 실행하는 US 일일 배치의 4단계 흐름이다.

단계작업스크립트소요 시간
1US 가격 수집 (FMP)collect_us_prices_fmp.py~10분
2US 기술적 지표run_batch.py --market US --step technical~2분
3US 밸류에이션 계산 (FMP + DB TTM)calc_us_valuations_fmp.py~45분
4스코어 캐시 갱신curl /api/cache/refresh백그라운드 ~5분
FMP 전환 (2026-03-27)
US 데이터 소스가 yfinance에서 FMP로 전면 교체되었다 (2026-03-27). FMP /stable/ 엔드포인트를 사용하며, PER/PBR은 DB 재무제표(TTM) + FMP 종가로 직접 계산한다. ROE/영업이익률/배당수익률은 FMP key-metrics-ttm/ratios-ttm에서 조회한다.

5. 데이터 소스 원칙

경고
DB에 데이터를 쓰는 작업은 반드시 확인 후 실행할 것. 과거 서버에서 잘못된 데이터를 덮어쓴 사고가 있었다. 특히 밸류에이션(PER/PBR/ROE) 데이터를 신뢰할 수 없는 소스로 덮어쓰면 전체 스코어링이 오염된다.
데이터신뢰 소스스크립트비고
KR 재무제표DART zip (직접 다운로드) + DART APIimport_dart_bulk.py / collectors/kr/financial.py분기 크론
KR 배당금DART 배당 공시 APIcollect_kr_dividends_dart_api.py연 1회 (사업보고서 후)
KR PER/PBR/ROEcalc_kr_valuations.py (DART 재무 + 종가)auto_daily.sh에서 매일 실행Naver 사용 안 함
KR 종가pykrx (KRX 공식)collectors/kr/price.py매일 크론
US 재무제표FMP (유료) + db/ zipcollectors/us/financial.py분기 크론
US 종가FMP /stable/quotecollect_us_prices_fmp.pySLA 보장, Starter 플랜
US 밸류에이션FMP + DB TTM 계산calc_us_valuations_fmp.pyPER/PBR: DB 재무+FMP종가, ROE/배당: FMP API
US 시가총액FMP /stable/quotecollect_us_prices_fmp.pystocks.market_cap 자동 업데이트
기술적 지표자체 계산 (종가 기반)calculators/technical.py매일 크론
원칙
  • KR 시가총액은 반드시 stocks.market_cap에서 로드 (valuations.market_cap 아님)
  • US 시가총액은 stocks.market_cap (FMP에서 매일 갱신) 또는 valuations.market_cap
  • 부채비율은 한국 표준(부채총계/자본총계) 사용. KR/US 동일 기준.
  • PER sanitize: <0.5 또는 >1000이면 None 처리
  • PBR sanitize: <0.05 또는 >200이면 None 처리

8. 코드 변경 시 체크리스트

코드를 수정하고 서버에 반영할 때 따르는 표준 절차이다.

  • 로컬에서 테스트 완료
  • git commit + git push
  • 서버에서 git pull origin main
  • 스코어링 변경인 경우: CACHE_SCHEMA_VERSION 올리기
  • 서버 재시작: systemctl restart stockdigging
  • 사이트 확인: stockdigging.com 접속하여 정상 동작 확인
절대 금지
  • 서버에서 직접 코드 수정 (항상 로컬 > push > pull)
  • UI만 바꿨는데 캐시 삭제 (CPU 과부하)
  • 크론 실행 중 수동 스크립트 실행 (SQLite 동시 쓰기 불가)

9. 스코어링 변경 시 체크리스트

scoring.py 또는 스코어링 로직을 변경할 때는 일반 코드 변경보다 추가 단계가 필요하다.

  • scoring.py 수정
  • quick_server.py에서 CACHE_SCHEMA_VERSION 증가 (예: "v8" > "v9")
  • 로컬 테스트: 캐시 삭제 후 서버 시작
# 로컬 테스트 (PowerShell)
Remove-Item -Force -ErrorAction SilentlyContinue data/score_cache.pkl, db/score_cache.pkl, score_cache.pkl
$env:PYTHONIOENCODING="utf-8"; python quick_server.py
# "캐시 완료: N개 조합" 메시지 확인
  • git push > 서버 git pull
  • 서버에서 캐시 삭제 + 재시작
# 서버
cd /var/www/stockdigging
rm -f data/score_cache.pkl && systemctl restart stockdigging
  • 캐시 자동 갱신 완료 대기 (약 5.5분)
  • 사이트에서 스코어 정상 표시 확인
CACHE_SCHEMA_VERSION

10. DB 관련 주의사항

핵심 원칙
  • DB 직접 수정 금지. 반드시 스크립트를 통해 수정한다.
  • 서버에서 데이터 수집 스크립트를 함부로 실행하지 말 것. 기존 데이터를 덮어쓸 수 있다.
