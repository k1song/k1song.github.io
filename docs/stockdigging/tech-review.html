2-2. 식별된 문제점

심각 — SQLite 2.1GB 단일 파일의 구조적 한계

SQLite는 WAL 모드에서도 동시 쓰기가 1개로 제한된다. stockpulse.db의 daily_prices 테이블이 520만 레코드를 담고 있어, 수집 스크립트가 실행되는 동안 API 응답 지연 또는 database is locked 에러가 발생할 수 있다.

또한 2.1GB 단일 파일은 백업 시 전체 복사가 필요하고, 네트워크 전송(SFTP)에 수십 분이 소요된다.

심각 — DB 6개 분산으로 트랜잭션 일관성 불가

예를 들어, 사용자(users.db)가 대회(competition.db)에 참가할 때 두 DB에 동시에 쓰기가 필요하지만, SQLite 간 트랜잭션은 불가능하다. 한쪽만 성공하면 데이터 불일치가 발생한다.

또한 stockpulse.db의 종목 데이터와 competition.db의 종목 선택을 조인하려면 애플리케이션 레벨에서 수동 조합해야 하며, 이는 코드 복잡도를 높인다.

개선 필요 — ORM/Raw SQL 이중 관리

stockpulse.db만 SQLAlchemy ORM을 사용하고, 나머지 5개 DB는 모두 sqlite3 모듈로 raw SQL을 직접 작성한다. 이는 유지보수 시 두 가지 패턴을 동시에 관리해야 하는 부담을 준다. 스키마 변경 시 ORM 마이그레이션과 raw SQL ALTER를 각각 별도로 수행해야 한다.

개선 필요 — valuations 테이블의 데이터 출처 혼재

valuations 테이블에는 두 종류의 데이터가 혼재한다:

  • 수집값: Naver Finance에서 직접 가져온 당일 PER/PBR/EPS/BPS (collectors/kr/valuation.py)
  • 계산값: calc_kr_valuations.py에서 financials 테이블 + 연말 주가로 재계산한 historical PER/PBR/ROE

동일 테이블에 출처가 다른 데이터가 섞이면, 특정 시점의 PER이 "라이브 수집"인지 "재계산"인지 구분이 불가능하다. 백테스트와 라이브 스코어링이 서로 다른 데이터를 참조할 위험이 있다.

개선 필요 — market_cap 이중 위치

market_cap 값이 두 곳에 존재한다:

  • KR 종목: stocks.market_cap에서 로드 (원화)
  • US 종목: valuations.market_cap에서 raw SQL로 별도 조회 (달러)

scoring.py의 _load_valuations()에서 KR일 때만 stocks 테이블을 추가 조회하는 분기 로직이 존재한다. ORM 모델에 market_cap 필드가 통일되어 있지 않아, 새 개발자가 어느 테이블을 참조해야 하는지 코드를 읽어야만 알 수 있다.

확인 필요 — arena.db와 competition.db 스키마 중복

arena.db(5개 테이블)와 competition.db(10개 테이블)가 유사한 스키마(시즌, 참가자, 종목 선택, 스냅샷)를 갖고 있으나 별도 파일로 관리된다. 기능적 차이가 충분히 크지 않다면 하나로 통합하는 것이 바람직하다.

확인 필요 — articles.db와 competition.db articles 중복

articles.db(1개 테이블)와 competition.db의 articles/article_likes/article_comments 테이블이 각각 별도로 존재한다. 기사 데이터가 두 곳에 분산되어 있으면 동기화 문제가 발생할 수 있다.

개선 필요 — US 데이터 의존성

US 데이터의 3대 축(가격, 밸류에이션, 재무)이 모두 yfinance에 의존한다. yfinance는 비공식 API이며:

  • Yahoo Finance의 웹 구조 변경 시 즉시 깨질 수 있음 (과거 3회 이상 발생)
  • Rate limiting이 예고 없이 변경됨
  • SLA/보장 없음, 상업적 사용에 대한 약관 모호

FMP API는 유료($79/년)이지만 무료 250콜/일 제한으로 6,700개 종목 전체를 커버하지 못한다. 상용 API(Polygon.io, Alpha Vantage Pro 등) 전환이 필요하다.

개선 필요 — KR 밸류에이션 2단계 계산의 복잡도

KR 밸류에이션은 2단계로 처리된다:

  1. calc_kr_valuations.py: DART 재무 데이터로 ROE, 부채비율, 영업이익률 계산
  2. merge_kr_valuations.py: Naver 수집값(PER/PBR)과 계산값(ROE 등)을 10개 필드로 병합

이 2단계 파이프라인은 실행 순서가 중요하며, 중간에 실패하면 부분적으로만 갱신된 데이터가 DB에 남을 수 있다. 하나의 스크립트로 통합하거나, 트랜잭션 보장이 필요하다.

개선 필요 — 생존편향 통제 불충분

stocks 테이블의 is_active 플래그만으로 상장폐지/합병/분할 종목을 관리하고 있다. 그러나:

  • 상장폐지 이전의 가격 데이터가 존재하는지 보장이 없음
  • 합병/분할 시 가격 소급 조정(adjusted close) 미적용 가능성
  • is_active가 "언제" 비활성화되었는지 타임스탬프 없음
  • 백테스트에서 "당시 활성 종목만" 필터링하는 메커니즘 부재

4-2. 식별된 문제점

개선 필요 — 1,907줄 단일 파일의 유지보수 부담

scoring.py에 다음이 모두 포함되어 있다:

  • 데이터 로딩 함수 (_load_valuations, _load_technical 등)
  • 30개 지표 계산 로직 (각 20~50줄)
  • 캘리브레이션 연동 (pct_score 호출)
  • KR/US 분기 로직 (시가총액, PER 기준 등)
  • 가격 목표(ATR) 계산 (STYLE_RISK 파라미터 18세트)
  • 캐시 관리 (pkl 저장/로드, 버전 체크)

하나의 지표를 수정하려면 1,907줄 파일 전체를 이해해야 한다. 단위 테스트도 불가능한 구조이다.

개선 필요 — 캐시 워밍업 5.5분 소요

warm_cache가 전체 종목(KR 2,663 + US 2,882)에 대해 6스타일 스코어를 일괄 계산한다:

  • KR: ~200초 (2,663 종목 x 6 스타일)
  • US: ~120초 (2,882 종목 x 6 스타일)
  • 합계: ~5.5분 (서버 재시작마다 발생)

이 시간 동안 API는 캐시 없이 응답하므로, 사용자에게 느린 응답이 제공된다. 종목 수가 증가하면 선형적으로 늘어나며, 10,000종목 시 약 11분이 소요될 것으로 예상된다.

개선 필요 — 캘리브레이션 이중 구조 (백분위 + 하드코딩 fallback)

상대 지표(PER, PBR, ROE, 영업이익률, 매출성장률)는 scoring_calibration.json의 백분위를 우선 사용하되, 값이 없으면 하드코딩 기준으로 fallback한다. 이는:

  • 새 지표 추가 시 "캘리브레이션에 추가했는가?"를 반드시 확인해야 함
  • fallback과 백분위의 결과가 다를 수 있어, 캘리브레이션 파일 유무에 따라 점수가 달라짐
  • KR/US 시장별 기준값이 다르므로 (예: PER 기준 KR 상위 2% vs US 상위 2%), 하드코딩 fallback이 어느 시장 기준인지 모호
개선 필요 — market_cap 로딩 비일관성

동일한 _load_valuations() 함수 내에서:

  • KR: ORM으로 stocks 테이블 조회 → raw SQL fallback
  • US: valuations ORM 모델에 market_cap 필드 없음 → raw SQL 직접 조회

이 비대칭은 "KR은 stocks.market_cap 사용, US는 valuations.market_cap 사용"이라는 규칙을 코드에 하드코딩하게 만들고, CLAUDE.md에 별도 경고로 기록해야 하는 지경에 이른다.

구조적 한계 — 종가 기반 스코어링

모든 지표가 전일 종가(daily_prices의 close)를 기반으로 계산된다. 장중 스코어링(intraday scoring)으로 전환하려면:

  • 실시간 데이터 피드 연동
  • 증분 계산 (변경된 종목만 재계산)
  • 캐시 무효화 전략 재설계
  • 기술적 지표 실시간 업데이트

이 전환은 근본적인 아키텍처 재설계를 의미한다. 현 단계에서는 종가 기반이 적절하나, 서비스 확장 시 고려해야 한다.

4-3. 긍정적 평가

잘 설계된 부분
  • ScoreDetail 구조체: (name, score 0~20, reason, sort_value) — 점수 + 사유 + 정렬값을 하나의 단위로 관리하는 설계는 디버깅과 UI 표시 모두에 유용하다.
  • 백분위 캘리브레이션: 시장 분포가 변할 때 점수가 자동 보정되는 시스템은 장기 운영에 적합하다. 매월 1일 자동 실행도 올바른 주기이다.
  • ATR 기반 가격 목표: 전략/기간별 18개 파라미터 세트로 변동성에 맞춘 동적 산출은 하드코딩 ±10%보다 훨씬 합리적이다.
  • PER/PBR sanitize: 극단값(PER<0.5, PER>1000, PBR<0.05, PBR>200)을 None 처리하는 방어 로직이 있다.
  • N/A 처리: score=-1 마커로 데이터 부재를 명시하고, _build_style_score에서 score>=0만 합산하는 방식은 건전하다.

5-2. 식별된 문제점

즉시 조치 — 비밀번호 하드코딩

upload_db.py에 서버 SSH 비밀번호가 평문으로 하드코딩되어 있다. 이 파일이 Git 저장소에 포함되어 있다면(확인 필요), 저장소에 접근 가능한 누구든 서버에 SSH 접속할 수 있다.

즉시 조치:

  • 비밀번호를 환경변수 또는 .env로 이동
  • 가능하면 SSH 키 인증으로 전환
  • git history에서 비밀번호 제거 (git filter-branch 또는 BFG)
즉시 조치 — auto_quarterly.sh 서버 미등록

DART 분기 재무제표를 수집하는 auto_quarterly.sh가 서버 크론탭에 등록되어 있지 않다(MEMORY.md 명시). 분기 재무제표 수집을 사람이 기억해서 수동 실행해야 하며, 누락 시 3개월간 오래된 재무 데이터로 스코어링이 수행된다.

즉시 조치 — root 계정 직접 사용

서버에 root 계정으로 직접 접속하여 서비스를 운영하고 있다. root 권한으로 실행 중인 프로세스가 취약점에 노출되면 전체 시스템이 위험하다. 또한 실수로 시스템 파일을 삭제/변경할 위험이 높다.

개선 필요 — 단일 장애점(SPOF)

전체 서비스가 Vultr VPS 1대에서 운영된다:

  • 서버 장애 시 전체 서비스 중단 (복구 시간 = 수동 대응 시간)
  • DB, API, 크론이 모두 같은 인스턴스에서 실행
  • 크론 수집 작업이 DB에 쓰기하는 동안 API 응답 지연 가능
  • Vultr 데이터센터 장애 시 복구 수단 없음
개선 필요 — CI/CD 부재

코드 배포 프로세스:

  1. 로컬에서 코드 수정
  2. SSH 접속
  3. git pull
  4. systemctl restart stockdigging
  5. (스코어링 변경 시) 캐시 삭제 + 5.5분 워밍업 대기

이 과정에서 테스트 실행 없이 프로덕션에 직접 배포되므로, 오류가 사용자에게 즉시 노출된다. 롤백도 수동(git revert + 재시작)이다.

개선 필요 — 모니터링/알림 부재

현재 시스템 상태를 확인할 수 있는 방법은 직접 SSH 접속뿐이다:

  • 크론 실패 시 알림 없음 → 데이터가 갱신되지 않아도 모름
  • 서버 다운 시 알림 없음 → 사용자 리포트로 인지
  • 디스크 사용량 모니터링 없음 → 2.1GB DB + 200+ zip이 디스크를 채울 수 있음
  • API 응답 시간 모니터링 없음 → 성능 저하 감지 불가
개선 필요 — 백업 전략 부재

upload_db.py가 DB 교체 전 .bak 파일을 생성하지만, 이는 자동 백업이 아닌 배포 시 부산물이다:

  • 정기 백업 스케줄 없음
  • 백업 파일의 외부 저장소 전송 없음
  • 복구 테스트 이력 없음
  • DB 손상 시 RPO(복구 시점 목표)가 "마지막 수동 배포 시점"
개선 필요 — 시크릿 관리

API 키(DART_API_KEY, FMP 키 등)가 .env 파일에 평문으로 저장되어 있다. 서버에 접근 가능한 누구든 모든 API 키를 볼 수 있다. Vault, AWS Secrets Manager, 또는 최소한 암호화된 환경변수 관리가 필요하다.

최종 평가

StockDigging의 기술 스택은 1인 개발자가 구축한 것치고 매우 높은 완성도를 보여준다. 6개 투자 스타일, 30개 지표, 9,400개 종목, 109개 API 엔드포인트, 크론 기반 자동 데이터 파이프라인은 상당한 실행력의 증거이다.

다만, 빠른 성장 과정에서 축적된 기술 부채(레거시 29개 스크립트, DART zip 200개, 6개 분산 DB)와 보안 취약점(비밀번호 하드코딩, root 사용)이 존재한다. 이는 초기 스타트업에서 흔히 볼 수 있는 패턴이며, P0 항목 6건을 즉시 처리하면 가장 큰 리스크를 해소할 수 있다.

P1~P2 항목을 순차 처리하면 3개월 내에 프로덕션 수준의 인프라로 전환할 수 있다.
