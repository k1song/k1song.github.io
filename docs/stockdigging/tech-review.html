0. 한 줄 요약

StockDigging v2는 v1의 "score_cache.pkl 단일 서버" 구조를 Next.js 16 + FastAPI + SQLite + precomputed JSON 정적 서빙 구조로 전면 재설계한 한국·미국 주식 스크리닝 랭킹 서비스다. 광고 수익 모델, 유료 회원 없음. 한국 투자자 타겟.

v2가 v1보다 나아진 점
  • P1 피팅/캘리브레이션 완전 제거 → raw 값 그대로 (신뢰성)
  • Next.js App Router → SSR SEO, i18n, OG 태그 자동화
  • deploy.py zero-downtime atomic swap
  • Supabase Auth (Google/Naver OAuth)
  • precomputed JSON → FastAPI 부하 0으로 랭킹 서빙
여전한 기술 부채
  • SQLite 동시성 — 크론 중 수동 작업 충돌 가능
  • 서버 단일 VPS — 단일 장애점
  • DB 자동 백업 미구축
  • KR Cash on hand ✅ 수집 완료 (2026-04-25 패치 — 금융주 "현금및예치금" 키워드 추가)

2-2. 시가총액 계산 (US ADR 이슈)

FMP의 valuations.market_cap은 ADR 230여 종목에서 현지 통화(TWD/CNY/EUR)로 저장되어 있어 랭킹을 오염시킨다. v2는 항상 실시간 계산을 사용한다.

# US 시총 계산 공식 (ADR 포함 전체)
US_market_cap = DailyPrice.close(USD) × Financial.shares_outstanding(최신 연간)

# 예: TSM
# DailyPrice.close = 168.32 USD
# shares_outstanding = 11,432,000,000
# → market_cap = 1.924T USD ✅  (FMP valuations.market_cap: TWD 기준 → 오염 ❌)
KR 시총
pykrx가 매일 stocks.market_cap을 원화로 업데이트. KR/US 통화 혼합 없음. KRW ↔ USD 변환은 FMP 환율 캐시 사용.

2-3. change_pct 계산 (US)

yfinance는 change_pct를 직접 제공하지 않는다. USPriceCollector가 10일 윈도우 다운로드 후 consecutive-close 방식으로 계산 후 upsert한다.

sym_df = df_raw[sym].sort_index()
closes = [fv(row["Close"]) for _, row in sym_df.iterrows()]
for i, (dt_idx, row) in enumerate(sym_df.iterrows()):
    cv = closes[i]
    prev = closes[i-1] if i > 0 else None
    chg = round((cv - prev) / prev * 100, 4) if prev and prev > 0 else None
    records.append({..., "change_pct": chg})

2-4. DST 자동 처리

US 크론은 고정 시각(20:10 / 21:10 UTC). 스크립트 내 _minutes_since_close()가 exchange_calendars로 실제 마감 시각을 계산 — 장 마감 전이면 자동 sys.exit(0). 서머타임 수동 전환 불필요.

3-2. 설계 원칙

Raw Data First — v2 핵심 원칙
24개 정렬 지표는 모두 원본 계산값. v1의 P1 피팅/백분위 변환 완전 폐기. 매일 재계산 공식:
  • PER = 당일 종가 × shares / TTM 순이익
  • PBR = 당일 종가 × shares / 순자산
  • EV/EBITDA = (시총 + total_debt - cash) / TTM EBITDA
  • FCF Yield = TTM FCF / 시총
  • 배당수익률 = TTM DPS / 당일 종가
SQLAlchemy 모델 동기화 필수
DB에 컬럼을 추가하면 backend/db/models.py에도 반드시 추가해야 한다. 없으면 setattr로 세팅해도 DB에 저장되지 않는다 (2026-04 실제 발생한 버그).

프론트엔드는 precomputed JSON을 우선 시도하고 없으면 FastAPI 폴백. 배치 완료 후 precompute 스크립트가 자동 재생성.

사용자 → Google/Naver OAuth → Supabase Auth ↓ JWT (localStorage 저장) ↓ Next.js API (/api/favorites) → Authorization: Bearer JWT ↓ Supabase REST API (RLS 적용) → user_id = auth.uid() 조건 강제
@supabase/ssr 사용 금지
chunked cookie 방식이 Next.js 미들웨어와 충돌, 로그인 세션이 불규칙하게 끊어지는 버그 (2026-04 실제 발생). supabase-js + localStorage + Bearer 패턴 고수.
  • favorites 테이블: Supabase PostgreSQL, RLS로 user 격리
  • 마이그레이션: supabase/migrations/001_favorites.sql → Supabase 대시보드 SQL Editor 실행
  • Google OAuth: Supabase 대시보드 > Auth > Providers > Google (client_id/secret 입력 필요)

7. 배포 파이프라인

로컬: git push → python deploy.py ↓ [서버] git pull + preflight (RAM/디스크) ↓ backend stop (RAM 확보), frontend 기존 .next로 계속 서빙 ↓ .next → .next.old 백업 ↓ npm run build ├─ 실패 → .next.old 복원 (다운타임 0) └─ 성공 ↓ systemctl restart --no-block (90초 타임아웃 회피) ↓ HTTP 헬스체크 (retry 3회) ├─ 실패 → .next.old 자동 롤백 └─ 성공 → .next.old 삭제 → ✅ Done
