4. 수용하는 비판

4-1. 법적 토대 부재

수용
사업자 정보 미공개, 유사투자자문업 미신고는 사실이다. 변명의 여지 없이 가장 먼저 정비해야 할 항목이다.

대응 계획:

  • 사이트 footer에 상호, 대표자, 사업자등록번호, 소재지, 연락처 공개
  • 유사투자자문업 신고 진행 (자본금 500만원 수준)
  • 이용약관 / 개인정보처리방침 페이지 작성

4-2. 데이터 투명성 부재

수용
DART, NAVER Finance, FMP 등 공식 소스를 사용하지만 사이트에 명시하지 않은 건 사실이다.

대응 계획:

  • 데이터 출처 명시 페이지 작성 (소스별 수집 항목, 갱신 주기)
  • 생존편향 통제 방법 (is_active 플래그) 고지
  • 백테스트 한계 (종가 기반, 슬리피지 미반영) 명시

4-3. 면책조항 부족

수용
현재 2문장 수준의 면책조항은 업계 기준에 미달한다.

대응 계획:

  • 별도 면책조항 페이지 작성 (투자 위험 고지, 과거 성과 ≠ 미래 수익)
  • Simply Wall St 수준의 상세 고지 참고

5. 반박하는 비판

5-1. "교과서 지표 조합, 파이썬 입문자도 주말에 가능"

반박 — 정보 부족에 기반한 과소평가
상대는 PER/PBR/RSI/MACD 나열만 보고 판단했다. 실제 시스템은 교과서 지표 나열과 근본적으로 다른 수준이다.

상대가 본 것: 지표 이름 목록 (PER, PBR, ROE, RSI, MACD)

실제 시스템:

  • 30개 지표 x 6스타일 매트릭스 — 지표별 커브 차별 적용 (sqrt, pow2, piecewise, zone, linear)
  • KR/US 시장별 독립 캘리브레이션 — p1=0점, p99=만점, 월간 자동 갱신
  • 중복 제거 실증 검증 — 상관계수(r) 기반 교차 검증. 베타 기각(r=0.49) → 매출 안정성으로 교체(r=0.07)
  • 49,590회 시뮬레이션으로 최적 파라미터 도출 (lookback, SMA 주기, 방어전략, 블렌딩 비율)
  • ATR 기반 동적 가격 목표 — 전략별, 종목별 변동성에 따라 개별 적용
  • 스코어링 v2 설계 — 5인 전문가 위원회 구성, 신규 6개 지표, 방식 변경 3개, 수식 변경 2개

"PER로 정렬한다"와 "30개 지표를 시장별 백분위로 캘리브레이션하고 상관관계를 검증하여 커브를 설계한다"는 완전히 다른 수준이다. 상대는 설계서를 본 적이 없으므로 이 비판은 정보 부족에서 비롯된 것이다.

상대 주장실제
"네이버 금융 수준"네이버 금융에 6전략 가중 스코어링, 투자성향 매칭, ATR 동적 목표가 없음
"교과서 1단계"캘리브레이션 + 상관관계 검증 + 시뮬레이션 최적화 = 팩터 모델 2~3단계 수준
"주말에 가능"스코어링 엔진 단독 1,907줄, 49,590회 시뮬레이션, 6단계 연구 파이프라인

5-2. "독자적 데이터 자산 없음"

반박 — 독자 자산의 정의를 좁게 적용
상대가 인정하는 독자 자산 예시(Simply Wall St의 DCF, TipRanks의 애널리스트 적중률)와 동일한 논리로 StockDigging도 독자 자산을 보유한다.
서비스독자 자산
Simply Wall StSnowflake 차트, DCF 기반 자체 밸류에이션
TipRanks애널리스트 적중률 추적 데이터
StockDigging6전략 30지표 스코어링 엔진 + 투자성향 진단 → 전략 매칭 + 적응형 벤치마크 실험 데이터 (301봇 x 3대회 x 95코호트 x 174실험)

Simply Wall St가 "Snowflake 차트"를 독자 자산이라 부르는 것과 동일한 논리로, 6전략 스코어링 + 성향 매칭 파이프라인은 충분히 독자 자산이다. 다만 이를 외부에 충분히 설명하지 않은 점은 인정하며, 스코어링 방법론 개괄 공개로 대응한다.

5-3. "바이브코딩 = 프로덕션 불가"

반박 — 현재 시스템 실태와 불일치
상대가 나열한 "바이브코딩으로 불가능한 것"은 이미 구현되어 운영 중이다.
상대 주장 "불가능"현재 StockDigging
실시간 데이터 파이프라인크론 6단계 자동 수집 — KR 매일(월~토), US 매일(월~금)
데이터 정합성PER/PBR sanitize (<0.5 or >1000 제거), 생존편향 is_active 관리
모니터링/알림배치 로그 + 이메일 알림 시스템
보안.env 분리, 비밀번호 해시, SQL parameterized query
백업/복구DB 백업 크론, gz 압축 보관
스케일링nginx 리버스 프록시 + 캐시 워밍업 (KR ~202초 + US ~120초)

규모 지표:

  • 109개 API 엔드포인트
  • KR 2,663개 + US 6,716개 종목 커버
  • 2,600+ 기술적 지표 레코드
  • 10,700회+ 시뮬레이션 수행
  • 분기별 자동 재무제표 수집 (DART API, FMP API)

완벽하지는 않다. CI/CD 파이프라인, 로드밸런싱 등은 아직 없다. 그러나 "프로토타입"이라 치부할 수준도 아니다. 라이브 서비스가 매일 자동으로 데이터를 수집·갱신하며 운영되고 있다.

6. 부분 수용하는 비판

6-1. "분석 깊이 무료 도구 미달"

부분 수용
MDD, 샤프비율, 벤치마크 비교가 시뮬레이션에 없는 것은 맞다. 그러나 "무료 도구 미달"은 과도한 표현이다.

수용: 시뮬레이션 결과에 리스크 지표(MDD, 샤프비율, 변동성, 벤치마크 비교)가 필요하다. 이미 1차 대응에서 수용했으며, 데이터는 계산 완료 상태로 표시 기능만 추가하면 된다.

반박: "무료 도구 미달"이라는 표현은 과도하다.

  • 네이버 금융에 6전략 가중 스코어링이 없다
  • 무료 도구에 투자성향 진단 → 전략 매칭 파이프라인이 없다
  • 무료 도구에 나만의 전략(30개 지표 가중치 커스텀)이 없다

리스크 지표 부재는 사실이지만, 서비스 전체가 무료 도구에 미달한다는 결론은 논리적 비약이다.

8. 결론

1차 대응으로 상대 비판의 근본 프레임(로보어드바이저 분류, 13~16억 비용, 투자일임업 5억)을 무너뜨렸다. 상대도 이를 인정하고 수정했다.

2차 비판에서 유지하는 7가지 중:

  • 3건 수용 — 법적 토대, 데이터 투명성, 면책조항. 모두 즉시~단기 해결 가능.
  • 3건 반박 — 기술력 과소평가, 독자 자산 부인, 프로덕션 부정. 모두 정보 부족에서 비롯된 비판.
  • 1건 부분 수용 — 리스크 지표 부재는 맞지만, 서비스 전체가 무료 도구 미달이라는 결론은 과도.

핵심 교훈: 수용할 비판(법적 토대 3종)을 빠르게 해결하면, 반박 가능한 비판은 자연히 설득력을 잃는다. 법적 토대가 없으면 기술력을 아무리 증명해도 "기본도 안 된 서비스"라는 프레임에서 벗어나기 어렵다.

다음 단계의 우선순위: 사업자 정보 공개 → 면책조항 → 유사투자자문업 신고 → 리스크 지표 추가 → 스코어링 방법론 공개
